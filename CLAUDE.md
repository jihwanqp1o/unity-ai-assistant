# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Unity AI Assistant — a 1-week MVP coding assistant for Unity developers. A PyQt5
always-on-top overlay lets a developer capture their Unity Editor screen on demand
(global hotkey or button), ask a question, and get an answer grounded in a small
curated set of Unity documentation snippets, generated by Claude (or a deterministic
mock when no API key is set).

## Commands

```powershell
# Setup
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt

# Unit tests (pure logic, no PyQt5/API key needed)
pytest tests\ -v
pytest tests\test_rag.py -v          # single file
pytest tests\test_rag.py::test_search_no_match_returns_empty -v  # single test

# Q6 success-metric check: RAG top-1 accuracy across data/test_scenarios.json
python scripts\run_scenario_eval.py
# Auto-switches to real Claude calls once ANTHROPIC_API_KEY is set (new terminal needed after setx)
setx ANTHROPIC_API_KEY "sk-ant-..."

# Run the full desktop app (PyQt window + global hotkey + real screen capture)
python main.py

# Standalone module checks (each has a __main__ demo)
python ui\overlay_window.py   # verify window renders + always-on-top on this OS
python core\hotkey.py         # verify Ctrl+Shift+C is detected (no admin needed on Windows)
python core\capture.py        # verify a real screenshot is captured

# Package as a single standalone .exe (no Python/venv needed to run it)
pip install pyinstaller
pyinstaller --onefile --windowed --name UnityAIAssistant --add-data "data;data" main.py
# --add-data is required: core/rag.py resolves data/unity_snippets.json relative to
# __file__, so the bundled data/ folder must sit next to the frozen code at the same
# relative path. Output: dist\UnityAIAssistant.exe
```

## Architecture

**Pipeline (wired in `main.py`'s `UnityAssistantApp`):**

```
hotkey/button → core/capture.py → core/rag.py → core/prompt_builder.py → core/claude_client.py → ui/overlay_window.py
```

1. **`core/capture.py`** (`ScreenCapture`) grabs the screen with `mss`, encodes to PNG via
   Pillow, and exposes `.to_base64()`. Capture and send are decoupled: capturing just
   stashes `pending_screenshot_b64` on the app; it's attached to the *next* question sent.
2. **`core/rag.py`** (`UnityDocRAG`) is a "light RAG" — no embeddings, no vector DB.
   It keyword-matches the query against `data/unity_snippets.json` (~33 curated Unity
   doc snippets). The scoring is the load-bearing part: keyword length drives the
   weight (`min(len(kw)*0.4, 4.0)` bonus) so a long, specific keyword like
   `collider-offset-mismatch` outranks a short generic one like "충돌" (collision).
   This weighting was added specifically to fix a 25%→100% accuracy regression — if
   you touch the scoring in `search()`, re-run `scripts/run_scenario_eval.py` against
   `data/test_scenarios.json` (each entry has an `expected_snippet_id`) to catch
   regressions.
3. **`core/prompt_builder.py`** assembles a single Anthropic Messages API user turn:
   image block first (if a screenshot is pending), then question + pasted code + RAG
   context as one text block. Pure functions, no `anthropic` import — safe to unit
   test without the SDK installed.
4. **`core/claude_client.py`** (`ClaudeClient`) auto-switches mock/real based solely on
   whether `ANTHROPIC_API_KEY` is set (read in `config.py`) — no code changes needed
   to flip modes. Mock mode never imports `anthropic`; it regex-extracts the question
   and first RAG doc title/pitfall from the assembled prompt and echoes back a
   templated response, so it validates pipeline wiring, not LLM answer quality.
5. **`ui/overlay_window.py`** (`OverlayWindow`) is a always-on-top PyQt5 widget
   (`WindowStaysOnTopHint`). It lazy-imports PyQt5 so the module is importable even
   where PyQt5 isn't installed. Status label color is keyword-matched against
   `_STATUS_COLORS` — the text passed to `set_status()` drives both the label text and
   its color, so new status strings should share a substring with an existing
   entry (or add one) to get sensible coloring. Code blocks are extracted with a single
   shared regex (`_CODE_BLOCK_RE`) and shown once in the code panel; the chat log gets
   the same text with the code fence replaced by a short pointer, to avoid duplicating
   the block twice on screen.

**Test layout:** `tests/conftest.py` inserts the repo root onto `sys.path`, so tests
import `core.*` directly without the package being installed. `tests/` only covers
`core/rag.py`, `core/prompt_builder.py`, and `core/claude_client.py` (mock mode) — all
pure-Python, no PyQt5/mss/keyboard dependency, so this suite runs anywhere.

**Windows-only, manually-verified modules** (no automated tests, verified by running
the `__main__` block directly): `core/hotkey.py` (global hotkey via `keyboard` — works
without admin rights in testing, contrary to the module's own docstring caveat),
`core/capture.py` (real `mss` screen grab), `ui/overlay_window.py` (real window
render + always-on-top). Verify these three manually after touching them, since there's
no headless test coverage for them.

**Known constraint:** the light RAG only knows what's curated into
`data/unity_snippets.json` — it has no fallback search for questions outside that set.
Expanding accuracy means adding/tuning snippets and keywords, not changing the matching
algorithm's shape.
