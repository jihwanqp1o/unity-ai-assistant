# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Unity AI Assistant — a coding assistant for Unity developers. A Unity developer captures
their Unity Editor screen on demand (global hotkey, via a small local agent), asks a question
in a web app, and gets an answer grounded in a small curated set of Unity documentation
snippets, generated by Gemini (or a deterministic mock when no API key is set).

**Originally a PyQt5 always-on-top desktop overlay** (single process, local-only). It was
migrated to a web architecture (2026-07-22) because the product needs to be a deployed,
multi-user web service. Browsers can't do arbitrary screen capture or OS-level global
hotkeys, so a small local capture agent still runs on the developer's machine — only the
UI moved to the web. See "Architecture" below for how the pieces fit together, and
`README.md`'s "웹 전환" section for the migration rationale.

## Commands

```powershell
# Setup
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt

# Unit tests (pure logic, no PyQt5/API key needed) — unchanged by the web migration
pytest tests\ -v
pytest tests\test_rag.py -v          # single file

# Backend tests (FastAPI TestClient, isolated SQLite temp file per test)
pytest backend\tests -v

# Both suites together
pytest tests\ backend\tests\ -v

# Q6 success-metric check: RAG top-1 accuracy across data/test_scenarios.json
python scripts\run_scenario_eval.py
# Auto-switches to real Gemini calls once GEMINI_API_KEY is set (new terminal needed after setx)
# Free key: https://aistudio.google.com/apikey
setx GEMINI_API_KEY "..."

# Run the backend (FastAPI)
uvicorn backend.app:app --reload --port 8000

# Run the frontend dev server (proxies /api to the backend, see frontend/vite.config.js)
cd frontend && npm install && npm run dev

# Run the local capture agent (Windows, real hotkey + real screen capture + tray icon)
python -m agent.local_agent
# First run opens a browser for login + device pairing

# Standalone module checks (each has a __main__ demo)
python core\rag.py            # verify RAG search output for a demo query
python core\prompt_builder.py # verify message assembly (no API key needed)
python core\llm_client.py     # verify mock mode end-to-end
python core\hotkey.py         # verify Ctrl+Shift+C is detected (no admin needed on Windows)
python core\capture.py        # verify a real screenshot is captured

# Build the deployable Docker image (React build + FastAPI serving it statically)
docker build -t unity-ai-assistant .
docker run -p 8000:8000 -e GEMINI_API_KEY=... -e JWT_SECRET=<32+ random bytes> unity-ai-assistant

# Build the local agent as a standalone Windows .exe + installer (see README's
# "로컬 캡처 에이전트 설치 프로그램 만들기" for the Inno Setup step)
pyinstaller --onefile --windowed --name UnityAIAssistantAgent agent_entry.py
```

## Architecture

**Two runtimes, one deployed web service:**

```
[local capture agent]                    [deployed backend: FastAPI]           [React frontend]
core/hotkey.py (detect hotkey)              backend/auth.py (signup/login,
core/capture.py (mss screenshot)  ──POST──▶   JWT cookie)
+ device pairing/token storage                backend/devices.py (pairing)
+ create session, upload screenshot            backend/sessions.py (session/
+ open default browser to session URL            screenshot/ask, calls core/*)  ◀─fetch/JSON─▶ login, pairing
                                                                                    approval, session
                                                                                    view, history
```

1. **`agent/`** (local capture agent, PyQt-free, a **system-tray-icon** background process,
   see `agent/local_agent.py`'s `LocalCaptureAgent`) — reuses `core/hotkey.py` and
   `core/capture.py` unchanged. Pairs with the backend once via a device-code flow
   (`agent/pairing.py`, same pattern as `gh auth login`): the agent gets a short
   `device_code`, opens the frontend's `/pair?code=...` page in the browser, the logged-in
   user approves it, and the agent polls until it receives a long-lived bearer token
   (stored via `agent/agent_config.py` in `%APPDATA%/UnityAIAssistant/agent_config.json`).
   On hotkey trigger (or the tray menu's "지금 캡처"), it captures a screenshot, then shows
   `agent/quick_dialog.py`'s native tkinter input box right there (over Unity, no browser
   switch) asking for the question. If answered, it `POST`s `/api/sessions/quick` (device
   auth) with screenshot + question in one call and gets the answer back immediately, then
   opens the browser straight to the already-answered session — one Unity↔browser round
   trip instead of two. If the dialog is cancelled, it falls back to the original flow
   (`POST /api/sessions` + `POST /api/sessions/{id}/screenshot`, question typed in the web
   UI). **The local agent never sees `GEMINI_API_KEY`** — only the backend does.
   - Uses `pystray` for the tray icon (status text as a disabled menu item, "지금 캡처",
     "히스토리 열기", "종료") since the process has no window and would otherwise be
     invisible to the user — it also fires a tray notification once pairing/hotkey setup
     finishes, since users couldn't otherwise tell the process had started. Pairing and
     hotkey registration happen in a background thread (`_bootstrap`) because
     `pystray.Icon.run()` blocks the main thread (a hard requirement on Windows) — see
     `LocalCaptureAgent.run()`.
   - `on_capture()` debounces repeated triggers within 1.5s (`_CAPTURE_DEBOUNCE_SECONDS`):
     `core/hotkey.py` used to fire on key-down, and holding Ctrl+Shift+C for even a moment
     triggers OS key-repeat, causing a double capture. Fixed at the source too —
     `keyboard.add_hotkey(..., trigger_on_release=True)` — but the debounce stays as a
     cheap safety net.
   - Because it can be packaged `--windowed` (no console, see below), `agent/pairing.py`'s
     `_log()` guards every `print()` behind an `if sys.stdout is not None` check — a frozen
     windowed exe has `sys.stdout is None`, and an unguarded `print()` would crash it.
   - **Self-updating** (`agent/updater.py`): on every startup (and via the tray's "업데이트
     확인"), it checks the GitHub Releases API and compares the release tag against
     `agent.__version__`. Only does anything when `sys.frozen` is set (i.e. running as the
     packaged exe, never for `python -m agent.local_agent` from source — there's no exe to
     replace). If newer, downloads the new exe to a temp dir and hands off to a generated
     `.bat` script that waits for this process's PID to exit, moves the new exe over
     `sys.executable`, and relaunches it — Windows won't let a running exe overwrite itself
     directly. The pairing token lives in `%APPDATA%` so it survives the swap; no
     re-approval needed. **Every new agent release must bump `agent/__init__.py`'s
     `__version__` to match the `agent-vX.Y.Z` git tag** — the updater compares those two
     directly, so forgetting the bump means that release is silently never detected as an
     update.
   - **Packaging**: `agent_entry.py` (repo root, not inside `agent/`) is the PyInstaller
     entry point — a thin wrapper so `agent.*`/`core.*`/`config` absolute imports resolve
     correctly once frozen (mirrors where the old `main.py` used to live). `installer/agent.iss`
     wraps the built `.exe` in an Inno Setup installer (Program Files install, Start Menu
     shortcut, optional "run at Windows startup" task). If you touch any docstring/comment
     in a `.py` file with a literal Windows path, watch out for `\U` (e.g. `dist\Unity...`)
     — Python parses `\U` as the start of an 8-hex-digit unicode escape and throws a
     `SyntaxError` at parse time, not just a display glitch; use forward slashes instead.
2. **`backend/`** (FastAPI, meant to be deployed, not run locally-only) —
   - `auth.py`: email/password signup+login, JWT stored in an httpOnly cookie
     (`backend/security.py` — pbkdf2 password hashing, no external hashing dependency).
   - `devices.py`: the pairing endpoints described above (`Device` model in `models.py`);
     agent requests authenticate via `Authorization: Bearer <token>` checked against a
     sha256 hash of the token (`get_current_device` dependency), never a raw comparison.
   - `sessions.py`: creates/reads/deletes capture sessions and — this is the load-bearing
     part — wires the question-answering flow using **`core/rag.py`, `core/prompt_builder.py`
     completely unchanged** from the desktop-app days, plus `core/llm_client.py` (Gemini,
     originally Claude — see below). `_answer_question()` does exactly what the old
     `main.py`'s `UnityAssistantApp.on_send()` used to do: `UnityDocRAG.search()` →
     `format_context()` → `build_messages()`/`build_system_prompt()` → `LLMClient.ask()` —
     it's shared by two endpoints: `/ask` (User-cookie auth, question typed in the web UI)
     and `/quick` (Device-token auth, screenshot+question submitted together by the agent's
     native quick-ask popup — see `agent/quick_dialog.py`). `DELETE /{session_id}`
     (User-cookie auth) removes a session outright.
   - `app.py`: mounts the routers, and if `frontend/dist` exists (i.e. `npm run build` was
     run), serves it as static files at `/` — so a single deployed process can serve both
     API and UI. The API routers register before the static mount, so `/api/*` always wins.
3. **`frontend/`** (React + Vite) — `src/pages/` (Login, Signup, Pair, Session, History),
   `src/components/` (ChatLog, CodePanel, StatusBadge), `src/lib/codeBlock.js` (a direct
   JS port of the old `ui/overlay_window.py`'s `_CODE_BLOCK_RE` extraction and
   `_STATUS_COLORS` keyword-matching — keep these two in sync if you touch either). Dev
   server proxies `/api/*` to the backend (`frontend/vite.config.js`) so cookies work
   same-origin without CORS wrangling; production serves both from the same FastAPI
   process instead (see `backend/app.py`).

### `core/` — the reused business logic

- **`core/rag.py`** (`UnityDocRAG`) is a "light RAG" — no embeddings, no vector DB. It
  keyword-matches the query against `data/unity_snippets.json` (~33 curated Unity doc
  snippets). The scoring is the load-bearing part: keyword length drives the weight
  (`min(len(kw)*0.4, 4.0)` bonus) so a long, specific keyword like
  `collider-offset-mismatch` outranks a short generic one like "충돌" (collision). This
  weighting was added specifically to fix a 25%→100% accuracy regression — if you touch
  the scoring in `search()`, re-run `scripts/run_scenario_eval.py` against
  `data/test_scenarios.json` (each entry has an `expected_snippet_id`) to catch
  regressions. Unchanged since the desktop app.
- **`core/prompt_builder.py`** assembles a single vendor-neutral user turn (role +
  text/image content blocks): image block first (if a screenshot is pending), then
  question + pasted code + RAG context as one text block. Pure functions, no LLM SDK
  import — safe to unit test without any SDK installed. Unchanged since the desktop app;
  `SYSTEM_PROMPT` was strengthened (2026-07-22) to explicitly refuse non-Unity questions
  and not leak its own instructions if asked about its identity.
- **`core/llm_client.py`** (`LLMClient`, originally `core/claude_client.py`/`ClaudeClient`
  — swapped from Anthropic to Gemini 2026-07-22 to use Gemini's free tier) auto-switches
  mock/real based solely on whether `GEMINI_API_KEY` is set (read directly from the
  environment) — no code changes needed to flip modes. It owns the translation from
  `prompt_builder.py`'s vendor-neutral content blocks into `google-genai`'s `contents`
  format (`_to_gemini_parts`) — if you swap vendors again, this is the only place that
  should need to change. Mock mode never imports `google-genai`; it regex-extracts the
  question and first RAG doc title/pitfall from the assembled prompt and echoes back a
  templated response, so it validates pipeline wiring, not LLM answer quality.
  `backend/sessions.py` keeps a module-level singleton of this (and of `UnityDocRAG`),
  mirroring how the old `main.py`'s `UnityAssistantApp.__init__` did it. Default model is
  `gemini-2.5-flash` (`config.GEMINI_MODEL`) — deliberately not the `-latest` alias, which
  Google can repoint without notice and has broken with 404s before. `max_tokens` defaults
  to 4096 (`config.GEMINI_MAX_OUTPUT_TOKENS`) — it was 1024 originally and answers were
  visibly getting cut off mid-sentence; if you see that again, this is the first thing to
  check.

**Test layout:** `tests/conftest.py` and `backend/tests/conftest.py` each insert the repo
root onto `sys.path`, so tests import `core.*`/`backend.*` directly without the packages
being installed. `tests/` covers `core/rag.py`, `core/prompt_builder.py`, and
`core/llm_client.py` (mock mode) — all pure-Python, no PyQt5/mss/keyboard dependency.
`backend/tests/test_flow.py` covers the full signup→pair→session→ask flow against an
isolated temp-file SQLite DB (see its `conftest.py` for how the DB is reset per test;
production uses Postgres — see the Known constraints below).

**Windows-only, manually-verified modules** (no automated tests, verified by running the
`__main__` block directly, or by running the agent end-to-end): `core/hotkey.py` (global
hotkey via `keyboard` — works without admin rights in testing, contrary to the module's
own docstring caveat), `core/capture.py` (real `mss` screen grab), and the whole
`agent/local_agent.py` flow (hotkey → capture → pairing/upload → browser open). Verify
these manually after touching them, since there's no headless test coverage for them.

**Known constraints:**
- The light RAG only knows what's curated into `data/unity_snippets.json` — it has no
  fallback search for questions outside that set. Expanding accuracy means adding/tuning
  snippets and keywords, not changing the matching algorithm's shape.
- `device_code` pairing has no expiry/single-use enforcement yet (MVP scope decision, see
  `backend/devices.py` docstring) — fine for a single-developer or small-team setup, needs
  hardening before wider rollout.
- Screenshots are stored as base64 text in the DB (`CaptureSession.screenshot_b64`) — fine
  at MVP scale, but should move to object storage if usage grows.
- Local dev defaults to a SQLite file (`backend/db.py`); the deployed backend uses Postgres
  (Render free tier, provisioned via `render.yaml`) — SQLite alone doesn't survive Render's
  free-tier disk being wiped on every restart/redeploy, which previously caused signup to
  always work while login to a previously-created account silently failed.
